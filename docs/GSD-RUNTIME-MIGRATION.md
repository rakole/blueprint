# GSD Runtime Migration

## Purpose And Scope

This document is the canonical runtime-porting reference for moving upstream GSD workflow runtime and hook behavior into Blueprint.

It covers only:
- retained-command workflow runtime for the 53 commands kept in Blueprint v1 planning
- retained Blueprint hook surfaces and explicitly omitted upstream-style hook surfaces

It does not reopen locked Blueprint architecture decisions. Omitted commands remain omitted unless a future decision update changes the retained-command contract.

## Mapping Rules And Legend

Allowed Blueprint destinations in this document:
- command spec owner in `docs/commands/`
- exact primary skill from `docs/SKILLS-AND-AGENTS.md`, resolved through `skills/<name>/SKILL.md`
- optional agent names already declared in the command specs and implemented as Gemini subagent files in `agents/*.md`
- exact MCP tool ids named in the command spec, or an existing MCP tool family when a spec does not name individual tools; command manifests may surface those tools through runtime FQNs such as `mcp__blueprint__...`
- advisory hooks already defined in `docs/HOOKS-POLICIES.md`
- explicit omission when Blueprint intentionally does not port a surface

Evidence tags used in the matrices:
- `locked`: the Blueprint destination is directly supported by current Blueprint decisions or command specs
- `spec-derived`: the statement is inferred from current Blueprint docs rather than a dedicated locked decision
- `needs-upstream-audit`: upstream GSD parity cannot be confirmed without inspecting the real upstream runtime source

Rules for using these tags here:
- Rows may carry more than one tag. A Blueprint destination can be `locked` while hook coverage remains `spec-derived` and upstream parity still `needs-upstream-audit`.
- The seven commands whose specs explicitly say GSD has no dedicated upstream workflow file are treated as Blueprint-native flow contracts and are not marked `needs-upstream-audit` for a missing workflow file.
- This document must not invent new Blueprint skills, MCP tools, agents, or hook surfaces.
- This matrix names canonical skill ids, agent ids, and tool ids; it does not use repo file paths as runtime handles.

## Phase 2.1 Closure Snapshot

The current drift-repair checkpoint locks the following Wave 0 runtime expectations:

- `/blu`, `help`, and `progress` must consult the runtime-aware command catalog and only recommend commands whose entry is `implemented`.
- `new-project` keeps MCP-owned bootstrap writes, and the `blueprint-bootstrap` skill now requires substantive bootstrap drafts, durable requirement traceability, and explicit brownfield next-step handling rather than bare scaffolding.
- `map-codebase` owns a seven-document bundle: `STACK.md`, `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, `TESTING.md`, `INTEGRATIONS.md`, and `CONCERNS.md`.
- Remaining explicit Blueprint deltas for shipped Wave 0 commands are limited to `/blu` naming, `.blueprint` state, Gemini-native routing, advisory hooks, no slash-command chaining, no repo-level hook config, and no self-update.
- Current bounded runtime restrictions that still need future review are explicit here: `new-project` now persists substantive bootstrap artifacts through the existing MCP tool surface, but it still does not claim a separate audited upstream research stage, and `map-codebase` still expects a Blueprint-initialized repo before writing into `.blueprint/codebase/`.

Wave 0 behaviors that remain narrower than upstream must be called out explicitly here rather than left implicit in command docs.

## Phase 2.2 Closure Snapshot

The closed Phase 2.2 checkpoint keeps future retained-command docs aligned without changing runtime exposure rules:

- Phase 2.1 and Phase 2.2 both completed on 2026-04-11.
- Phase 3 discovery shipped on 2026-04-11 and remains in parity closeout while the later shipped lifecycle, governance, and roadmap-admin slices keep their current contracts.
- `plan-phase` and `execute-phase` are now implemented in Blueprint on top of the plan and summary MCP substrates, and the validation slice now ships through `validate-phase` and `verify-work`. `execute-phase` also reads the canonical `phase.summary` contract before summary writes, uses `blueprint_phase_execution_targets` for deterministic selection, and keeps later-wave work blocked until any lower-wave pending plans are closed.
- `add-phase`, `insert-phase`, `remove-phase`, `plan-milestone-gaps`, `audit-milestone`, `complete-milestone`, `milestone-summary`, `new-milestone`, and `list-phase-assumptions` are now implemented in Blueprint as the shipped Wave 2 roadmap, milestone, and discovery commands: `add-phase` appends the next whole-number phase, scaffolds the matching phase directory, and updates state; `insert-phase` inserts the next decimal phase after an existing integer phase without renumbering later roadmap entries, scaffolds the matching phase directory, and routes back into discovery; `remove-phase` previews the target phase through phase location, requires explicit confirmation through Gemini CLI `ask_user`, removes a future phase, deletes the matching phase directory, and renumbers later roadmap references plus phase artifacts; `plan-milestone-gaps` groups audit findings into a coherent gap-closure slice; `audit-milestone` compares milestone intent against completed evidence and writes a durable report; `complete-milestone` and `milestone-summary` persist durable closeout reports on the existing artifact/state substrate; `new-milestone` starts the next milestone through a carry-forward scaffold flow that preserves historical phase artifacts and phase-number continuity; and `list-phase-assumptions` surfaces phase assumptions before planning.
- `secure-phase` is now implemented as the first shipped Wave 4 review-family command, using `blueprint_review_record` plus the new `blueprint-review` skill and `blueprint-security-auditor` contract to persist `XX-SECURITY.md` without widening the rest of the review surface early.
- Future command metadata must stay consistent across `docs/COMMAND-CATALOG.md`, `docs/SKILLS-AND-AGENTS.md`, this migration matrix, and the per-command specs in `docs/commands/`.
- `next` and `do` remain owned by `blueprint-router`.
- `pause-work` and `resume-work` remain owned by `blueprint-governance`.
- `plan-milestone-gaps` remains owned by `blueprint-roadmap-admin`.
- High-risk retained flows such as `quick`, `code-review-fix`, `audit-fix`, `ship`, and `undo`, plus planned flows such as `new-workspace`, `remove-workspace`, and `reapply-patches`, must continue to name their bounded-agent and MCP-owned persistence contracts explicitly; do not flatten them into command-only prose.
- Phase 2.2 did not change `blueprint_command_catalog` status semantics or make blocked commands routable.
- Closing Phase 2.2 does not make documented future commands runnable; runtime exposure still depends on real manifests, primary skills, and required MCP tools.
