# Blueprint MCP Tools

## Purpose

Blueprint commands use MCP tools for deterministic state operations. This keeps filesystem contracts, validation, and path safety consistent across commands and prevents prompt-only state mutation from becoming the real runtime.

## Design Rules

- Tools own persistence.
- Commands own UX.
- Skills own orchestration.
- Agents own bounded deep work.
- Shared runtime security lives below the tool surface and is consumed by MCP tools rather than by command-only prompt logic.
- Every registered tool returns the same result object in both MCP `structuredContent` and MCP `content[0].text`; the text form is compact JSON so hosts that expose only `content` still give the model the full deterministic result.
- Failed mutating tool calls should append a best-effort diagnostic entry to `.blueprint/mcp-write-failures.ndjson` before the rejection or exception is surfaced back to the model.

## Current Registered Tool Surface

These are the tool names actually registered by `src/mcp/server.ts` today. Future command specs may reference planned tools, but those are not part of the live runtime until they are registered here.

## Read-Only MCP Resource Contract

Blueprint now exposes the live read-only command resources and keeps the remaining resource contract documented for later grounding surfaces. `blueprint://commands/catalog` is live today. `blueprint://commands/<command>/runtime-contract` is also live today, but only for commands whose catalog entry is `implemented`. The phase, codebase, and reports resource views below are still planned, so commands must keep using the existing local docs plus read-oriented MCP tools directly when a resource is unavailable.

Blueprint resources are read-only discovery surfaces. They do not accept write inputs, they do not become a second persistence layer, and they must not move writes off the existing MCP tool surface.

| Resource URI | Purpose | Backing truth | Write policy | Fallback when unavailable |
|---|---|---|---|---|
| `blueprint://commands/catalog` | Read the retained runtime command catalog as one resource view | live `blueprint_command_catalog` truth | read-only; never a write target | call `blueprint_command_catalog` directly if the resource is unavailable |
| `blueprint://commands/<command>/runtime-contract` | Read the runtime contract for one retained implemented command | live command-catalog metadata plus the locked command spec and runtime-reference row for that implemented command | read-only; never a write target | read the command spec, runtime reference, and current tool docs directly when the resource is unavailable |
| `blueprint://phases/<phase>/bundle` | Read a phase-grounding bundle for discovery-style workflows | saved `.blueprint/` core docs plus the resolved phase artifact set | read-only; never a write target | use `blueprint_phase_context`, artifact reads, roadmap/state reads, and local docs directly |
| `blueprint://codebase/bundle` | Read the saved seven-document codebase mapping bundle as one resource view | `.blueprint/codebase/*.md` plus existing artifact-contract truth | read-only; never a write target | use `blueprint_artifact_list`, `blueprint_artifact_contract_read`, and local files directly |
| `blueprint://reports/latest` | Read the latest-report index for durable Blueprint reports | `.blueprint/reports/` inventory and existing report metadata | read-only; never a write target | use `blueprint_artifact_list` and direct report reads |

Resource-adoption guardrails:

- Prefer `list_mcp_resources` and `read_mcp_resource` for the live command resources today, and for the remaining resource views only after those resources exist.
- Router, progress, and discovery-style reads may adopt these resources later, but implemented-only command exposure must stay unchanged.
- Resource views must mirror existing command/tool truth; they do not get to invent new routing, status, or persistence semantics.

## Model-Facing Call Contracts

These notes are the shared prompt-facing contract for the current runtime. Command manifests, skills, agents, and command docs should follow them when they tell the model how to call Blueprint MCP tools safely.

### Numeric Phase And Artifact IDs

- Fields such as `phase`, `after`, and `planId` accept only numeric Blueprint references or numeric artifact ids.
- Omit `phase` only when a command intentionally wants the runtime to infer the current phase from `STATE.md` or the roadmap.
- Never pass phase directories, slugs, filenames, combined tokens such as `02-01`, or artifact frontmatter keys where the tool expects a numeric id.
- Treat returned `phaseNumber`, `phasePrefix`, `planId`, `phaseDir`, `path`, and similar fields as authoritative. Do not derive replacement ids or paths from chat text when the tool already returned the canonical value.

### Capture Index Mutations

- `blueprint_artifact_mutate_index` defaults to append mode when `action` is omitted. Append flows should pass `target` plus `entry.text`.
- Use only the returned `createdEntryIds`, `duplicateEntryIds`, `matchedEntryIds`, and `reservedPhase` fields as authoritative capture identifiers or reserved stub metadata. Do not invent `NOTE-*`, `TODO-*`, or `BACKLOG-*` ids manually.
- `reservePhaseStub` applies only to backlog append requests. When a reserved stub is created, the returned `reservedPhase.phaseNumber`, `reservedPhase.phaseDir`, and `reservedPhase.artifactPaths` are authoritative.
- Todo status changes must use `action: "update"` plus either `match.id` or exact `match.text`, and they must include `update.status`.
- Backlog or batch status changes should use `updates[]` with canonical entry ids from prior tool output, and `reservedPhase: null` is the supported way to clear consumed reservation metadata.

### Backlog Promotion

- `blueprint_roadmap_promote_backlog` should be called with `previewOnly: true`, or with no `backlogIds`, when the command wants a deterministic preview without mutation.
- Promotion calls should reuse only confirmed `backlogIds` returned by the preview result.
- Treat returned `promotedItems`, `selectedBacklogIds`, and `createdPhaseDirs` as authoritative. Do not invent promoted phase numbers or reserved-stub reuse paths manually.

### Scaffolding

- `blueprint_artifact_scaffold` accepts only supported repo-relative Blueprint artifact paths.
- `blueprint_phase_artifact_scaffold` accepts numeric `phase`, artifact enum, and optional `overwrite`, then resolves the canonical phase artifact path through MCP-owned phase location. Use it when a phase command needs a starter placeholder without reconstructing phase directories or filenames.
- `blueprint_artifact_contract_read` returns the runtime-owned canonical authoring contract object for a given artifact id such as `phase.research`, `phase.verification`, or `report.debug`; `contract.authoringTemplate` is present for Markdown-authored contracts and compatibility/preview surfaces such as `phase.plan`, while model-only public reads such as `phase.context`, `phase.verification`, `phase.uat`, and `report.add-tests` expose `contract.modelContract` without a public authoring template. If runtime contracts expose a separate scaffold template in the future, keep that starter material distinct from the final authoring authority.
- For `bootstrap.roadmap`, `contract.modelContract` is the ROADMAP parity authority: it exposes `schemaPath`, `jsonSchema`, `minimalValidExample`, `renderedHeadings`, `qualityRules`, and `contextBindings` for milestone, bootstrap status, requirement coverage, phases, optional phase details, phase status vocabulary, dependencies, inserted marker, durable requirement IDs, and 2-5 success criteria.
- Do not pass bare artifact names such as `STACK`, absolute filesystem paths, or ad hoc report filenames.
- Use scaffolding only for first-write seeding or template regeneration. Do not treat scaffold output as the final persistent content for filled-in research, context, UI-spec, plan, summary, validation, or report artifacts, and do not preserve literal scaffold placeholders or example filler in final writes.
- Treat returned `createdFiles` and `reusedFiles` as authoritative for what the tool actually touched.
- For `/blu-new-milestone`, also treat returned `highestBasePhaseNumber`, `firstPhaseNumber`, `firstPhasePrefix`, `firstPhaseDir`, `firstContextPath`, `deletedPhaseDirectories`, and `renamedPhaseDirectories` as the MCP receipt for the first carried-forward phase; do not recompute or override them in prompt text.
- Blueprint does not manage a repository-root `CONTEXT.md`. Brownfield mapping scaffolds and writes only `.blueprint/codebase/*.md`; phase context scaffolding and writes are limited to `.blueprint/phases/<phase>/<XX>-CONTEXT.md`.

### Roadmap Creation And Insertion

- For plain `/blu-add-phase`, call `blueprint_roadmap_add_phase` with `description`, confirmed durable `requirementIds`, concrete `goal`, 2-5 item `successCriteria`, and the confirmed `expectedPhaseNumber` stale guard after reading the roadmap and confirming the preview. `cwd` is optional. Audit-backed callers may additionally pass `auditBackedDetails` only when a documented audit-follow-up workflow supplies that evidence; `auditBackedDetails.repairRequirementIds` remains the repair traceability path for audit-backed requirement reassignment.
- `blueprint_roadmap_insert_phase` accepts an integer anchor in `after` plus `description`, concrete `goal`, 2-5 item `successCriteria`, and confirmed durable `requirementIds` declared in `.blueprint/REQUIREMENTS.md`; it maps those requirement table rows to the inserted phase and rejects `none yet`, placeholder, blank, or undeclared requirement mappings.
- Do not precompute decimal suffixes, slugs, or phase directories in prompt logic. For add-phase, the command may compute the previewed next integer from `blueprint_roadmap_read` solely for confirmation and `expectedPhaseNumber`; the tool remains authoritative for the committed phase metadata.
- After the tool returns, use the returned `afterPhaseNumber`, `phaseNumber`, `phasePrefix`, and `phaseDir` as authoritative when follow-on scaffolding or routing needs a concrete phase target.
- For `/blu-insert-phase`, `blueprint_artifact_scaffold` seeds only `${phaseDir}/${phasePrefix}-CONTEXT.md`; rich phase context authoring stays in `/blu-discuss-phase` against the canonical `phase.context` contract.

### Phase Artifact Writes And Checkpoints

- `blueprint_phase_artifact_write` accepts numeric `phase`, enum `artifact`, and exactly one input. For `artifact: "context"`, pass a structured `phase.context` `model`; Markdown `content` fallback is rejected. For `discussion-log`, `research`, and `ui-spec`, pass full Markdown `content`.
- `blueprint_phase_ui_skip_write` accepts numeric `phase` plus one `skipRationale` string and renders the minimal valid explicit-skip `XX-UI-SPEC.md` shape. Use it when the run is skip mode instead of loading the full `phase.ui-spec` authoring template.
- Do not pass artifact filenames, `phaseDir`, or `phasePrefix` into `blueprint_phase_artifact_write`; the tool owns the artifact path.
- `/blu-discuss-phase` owns authoring and repair of `artifact: "context"`. `/blu-research-phase`, `/blu-ui-phase`, and `/blu-plan-phase` may read `XX-CONTEXT.md` and route back to `/blu-discuss-phase <phase>` when it is missing, invalid, or unusable; they must not repair, overwrite, or synthesize context themselves.
- When normalizing authored phase artifacts, prefer `blueprint_artifact_contract_read` over copied prompt-local templates. Use scaffold output only as disposable starter seed; final saved artifacts must follow the live authoring template and replace scaffold literals with real content.
- Research writes validate in `strict` mode by default. Use `validationMode: "warn"` only when the command intentionally wants warnings without blocking the write attempt.
- Research validation warnings can include `diagnostics` with `severity: "warning"`. Warning diagnostics do not make `validation.valid=false`, but commands should report or repair them when they affect planner-critical recommendations.
- Research writes should be normalized to Blueprint's exact `XX-RESEARCH.md` template before calling the tool, and angle-bracket placeholders must be replaced with real content.
- Research authoring should preserve repository search discipline from the research-phase runtime contract: use saved context and codebase summaries before broad reads, prefer `rg --files` plus path filters and scoped content searches, record per-strand search notes, treat semantic/navigation packets as optional parent-owned evidence, and treat remote code-search hits as hints until local repo evidence confirms them. New research should also preserve claim-addressable provenance for planner-critical claims; missing provenance rows in older otherwise-valid research produce warnings unless stricter validation is introduced later.
- Validation repair loops are bounded. After validation diagnostics, repair the same draft once and retry the same MCP path; if the retry returns identical diagnostics, stop, preserve any command checkpoint, report the exact diagnostics plus the next safe action, and do not inspect MCP source files as a repair strategy.
- `blueprint_phase_checkpoint_get` accepts optional `expectedOwnerCommand` and `expectedMode` fields so commands can deterministically avoid resuming stale or foreign continuation state. Legacy saved checkpoints remain readable as non-resumable evidence, but reads include `warnings` and `safeToResume: false`.
- `blueprint_phase_checkpoint_put` requires checkpoint v2: `schemaVersion: 2`, `ownerCommand`, and top-level `mode`. Discuss checkpoints must include `progress`, `areaQueue`, `carryForward`, and `readSet`; research checkpoints must include `researchLedger.schemaVersion: "research-ledger/v1"` and `researchLedger.strands`. The tool owns the shared checkpoint filename and location, and rejects overwriting a checkpoint owned by a different command, mode, or legacy shape.
- `blueprint_phase_checkpoint_delete` accepts the same `expectedOwnerCommand` and `expectedMode` guard fields. When a checkpoint exists, the tool refuses unguarded deletes and refuses to delete a foreign shared checkpoint, reporting the ownership/mode blocker in `reason`.
- Treat returned `path`, `written`, `created`, `overwritten`, and `status` fields as authoritative for artifact and checkpoint persistence.

### Plan, Summary, And Validation Artifacts

- `blueprint_phase_plan_write` omits `planId` to auto-assign the next plan slot. When targeting an existing plan, pass only the numeric plan id.
- `/blu-plan-phase` reads `blueprint_phase_plan_authoring_context`, authors against its `taskSchema`, re-reads that authoring context immediately before each model validation/write after any successful plan write because saved plan files are intentional later-slot evidence artifacts, validates the structured model through `blueprint_phase_plan_validate_model`, writes the same model through `blueprint_phase_plan_write` with `validationMode: "strict"` and `authoringMode: "model-only"`, and follows it with `blueprint_phase_plan_validate`; warn-mode plan writes and Markdown `content` fallback are reserved for lower-level compatibility or explicit repair/migration exceptions outside `/blu-plan-phase`.
- `blueprint_phase_plan_validate_model` runs a best-effort prospective saved-plan preflight against its internal rendered preview before trimming the standalone response; dependency, cycle, and wave-order coherence can surface as diagnostics there, while roadmap coverage stays warning-level during incremental authoring.
- Successful incremental `blueprint_phase_plan_write` calls can still return top-level `warnings` and nested `validation.warnings` when the prospective saved plan set remains incomplete. Callers must inspect those warnings and run `blueprint_phase_plan_validate` before treating planning as complete, even when `validation.valid === true`.
- Final roadmap coverage for `blueprint_phase_plan_validate` comes from saved plan frontmatter `requirements`, not from `Requirement Coverage` table rows.
- `blueprint_phase_plan_index` and `blueprint_phase_plan_read` surface an explicit `gapClosure` signal from optional `gap_closure: true` plan frontmatter. `--gaps-only` should target those plans rather than inferring gap closure from missing summaries alone.
- `blueprint_phase_execution_targets` is the deterministic read path for `/blu-execute-phase` target selection. It derives candidate and selected plan ids for the default case plus `--wave` and `--gaps-only`, keeps any lower-wave pending plan as an absolute blocker for later-wave work, including combined `--gaps-only --wave` runs, surfaces stale or invalid selected-plan blockers from the current plan index, and returns overwrite candidates plus overlap warnings for selected or overlapping plan ids.
- `blueprint_phase_summary_write` requires numeric `phase`, numeric `planId`, Markdown summary `content`, and an explicit `Status` marker for new writes. The matching plan must already exist; formatting drift is warning-level, but empty content, invalid status, failed verification contradictions, incomplete dependencies on `COMPLETED`, and invalid linked plans still block completion evidence.
- `/blu-execute-phase` reads `blueprint_phase_summary_authoring_context`, drafts Markdown summary `content` against the runtime contract, validates that draft through `blueprint_phase_summary_validate_model`, writes the same Markdown content through `blueprint_phase_summary_write`, and then reruns summary index plus artifact validation before syncing state.
- `blueprint_phase_summary_write` refuses to write execution evidence for an invalid linked plan. Repair invalid plan frontmatter, missing dependencies, or stale plan metadata before summarizing implementation work.
- `/blu-execute-phase` should call `blueprint_state_update` with `base: "synced"` after rerunning summary index and artifact validation so `STATE.md` reflects refreshed summary truth rather than stale stored state.
- `blueprint_phase_summary_index` treats only valid `COMPLETED` summaries as completed plans. Valid `PARTIAL` or `BLOCKED` summaries stay in `pendingPlans` so execute-phase can resume or repair honestly.
- `blueprint_phase_validation_write` requires numeric `phase`, enum `artifact` (`verification` or `uat`), and exactly one of full validation `content` or a structured `model`. For `phase.verification` and `phase.uat`, `blueprint_phase_validation_validate_model` owns schema-first model validation; `/blu-validate-phase` and `/blu-verify-work` writes must use `authoringMode: "model-only"`. Model writes reject prompt-owned identity keys, copied minimal-example phrases, unsupported fields, and missing required model ledgers before rendering canonical Markdown.
- Verification and UAT writes both require saved execution summaries. UAT also requires an existing verification artifact.
- `validate-phase` and `add-tests` should read `blueprint_phase_validation_authoring_context` and `blueprint_artifact_contract_read` for `phase.verification`, build a structured verification payload, and keep every completed summary path inside the schema-owned evidence ledger. For `validate-phase`, the authoring authority is `contract.modelContract` plus `blueprint_phase_validation_authoring_context.taskSchema`, and `blueprint_phase_validation_validate_model.renderPreview` is the canonical preview; the public `phase.verification` contract read does not expose `authoringTemplate`. `validate-phase` must call `blueprint_phase_validation_validate_model` and persist the same model with `authoringMode: "model-only"`; add-tests may keep using render/write compatibility when it is extending existing verification evidence. `validate-phase` must author the `phase.verification` 1.1.0 model with `status` equal to `gateState`, normalize lowercase `covered` coverage to `COVERED`, preserve `sessionState`, `checkpoint`, `testMatrix`, `resultSummary`, `observedBehavior`, `unresolvedGaps`, `structuredGaps`, and `followUpFixes`, and use MCP `content.text` surfaces instead of shell reads when a host hides structured content or generic file reads ignore `.blueprint/`; all registered MCP tool text surfaces mirror `structuredContent` as compact JSON. `validate-phase` must fill the richer verification sections for requirement/task coverage, evidence metadata, manual-only or deferred coverage, gate state, gap classification, gaps found, suggested repairs, and next safe action. `add-tests` must also read `blueprint_artifact_contract_read` for `report.add-tests`, then `blueprint_artifact_report_authoring_context`; for the durable report, the authoring authority is `contract.modelContract` plus `blueprint_artifact_report_authoring_context.taskSchema`, and `blueprint_artifact_report_validate_model.renderPreview` is the canonical preview because the public `report.add-tests` contract read does not expose `authoringTemplate`. `add-tests` must validate the structured report model with `blueprint_artifact_report_validate_model` and write the same model through `blueprint_artifact_report_write`; the model includes classification, approved test plan, tests added or updated, targeted commands, generated/passing/failing/blocked counts, bugs or blockers, verification write status, report write status, remaining gaps, and next safe action.
- `verify-work` should read `blueprint_phase_validation_authoring_context` and `blueprint_artifact_contract_read` for `phase.uat`, build a structured UAT payload against `contract.modelContract` plus the returned `taskSchema`, use `blueprint_phase_validation_validate_model.renderPreview` as the canonical preview, and then call `blueprint_phase_validation_write` with the same structured `model` and `authoringMode: "model-only"`. The public `phase.uat` contract read does not expose `authoringTemplate`. New or updated UAT content must include saved summary citations plus current test state, test matrix, result counts, blocked-prerequisite separation, structured gaps with inferred severity, in-artifact checkpoint state, and retry-on-invalid behavior through MCP rather than direct file edits. User-reported issues and blocked prerequisites are UAT evidence; only explicit follow-up-fix capture needs a separate confirmation gate.
- Treat returned `path`, `linkedPlanPath`, `summaryPaths`, `planId`, `gapClosurePlans`, and `status` as authoritative. Do not invent plan or summary filenames manually.
- Validation reads and writes expose only summaries whose `**Plan:**` marker matches the linked plan path; invalid summaries stay out of `summaryPaths`.

### Config, Bootstrap, And Pause Handoff

- `blueprint_config_get` defaults `scope` to `effective`. `defaultsPath` is only an override for the host-global defaults file.

- Use `blueprint_config_set` with `scope: "defaults"` only when the user explicitly wants saved-default changes.
- Use `blueprint_config_set_profile` instead of a generic config patch when the only intended mutation is `model_profile`.

## Path Safety Rules

- All tools operate relative to the repo root or the locked host-global Blueprint directory.
- Config tools may read or write only `.blueprint/config.json` and `~/.<host>/blueprint/defaults.json`; they must not create ad hoc config files elsewhere.
- Tools must reject path traversal, absolute-path misuse for repo-relative inputs, null bytes, and symlink escapes.
- Tools should use shared safe parsing with size limits for config, checkpoint, and registry-style JSON inputs.
- Prompt-boundary-sensitive writes such as reports, phase artifacts, review artifacts, pause handoffs, and capture indexes should be checked through the shared security layer before persistence.
- Tools must not write into the installed extension directory as part of normal command execution.
