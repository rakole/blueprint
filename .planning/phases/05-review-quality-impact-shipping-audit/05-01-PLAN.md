# Phase 5 Plan 01: Review Artifact Quality

## Goal

Audit Blueprint review artifact quality for `code-review`, `secure-phase`, `review`, and `ui-review`, with emphasis on schema-plus-evidence rigor, saved artifact parsing, residual quality checks, model-only persistence, and implemented-only next-action routing.

## Scope

In scope:
- `review.code-review`, `review.security`, `review.peer-review`, and `review.ui-review` contracts.
- Command docs and TOML manifests for `/blu-code-review`, `/blu-secure-phase`, `/blu-review`, and `/blu-ui-review`.
- `skills/blueprint-review/` shared orchestration and command-specific runtime references.
- `src/mcp/tools/review.ts`, review artifact schemas, rendered artifact validation, and focused review tests.

Out of scope:
- Applying review findings or source fixes.
- Remediation commands (`code-review-fix`, `audit-fix`) except as next-action targets.
- Live security or UI audit probes that would mutate repo state.

#### Action

Verify that each reviewed command requires these concrete target states:

- `/blu-code-review` reads `review.code-review`, resolves exact files through `blueprint_review_scope`, validates model fields `verdict`, `reviewSummary`, `positiveSignals`, `findings`, `evidenceCoverage`, `followUps`, and `nextSafeAction`, then persists through `blueprint_review_record` with Markdown `content` rejected.
- `/blu-secure-phase` stays threat-model-bound, requires completed summary evidence, validates `review.security`, blocks pending plans before persistence, and avoids widening into a generic security scan.
- `/blu-review` creates peer-review evidence from selected phase plans and saved evidence, preserves reviewer disagreement, validates `review.peer-review`, and routes only through implemented commands.
- `/blu-ui-review` uses saved UI/summary/evidence context, records repo-line, screenshot, or visual-observation evidence, validates `review.ui-review`, and preserves visual-evidence limitations.

- Code-review findings include scoped repo-relative file or file:line evidence, impact, and concrete recommendation.

- Peer review records selected plans, reviewer status, disagreement or unavailable reviewer evidence, and next-safe-action truth tables.
- UI review rejects fabricated repo-line evidence, distinguishes screenshot and visual observations, and blocks PASS when pending plans or unresolved UI issues remain.

## Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| T-01 | The audit accepts schema-valid but evidence-weak review artifacts. | Compare schema checks, residual validation, rendered output, and tests against D-01 through D-03 before closing. |
| T-02 | The audit starts fixing review findings. | Treat findings as evidence only; allowed writes are bug docs, index rows, and summaries. |
| T-03 | Review routing recommends planned-only commands. | Check next-action values against implemented command routing in docs/runtime/tests. |

## Must Haves

- Review artifacts must be audited against both schema and evidence quality.
- The shared review substrate must be traced through docs, manifests, skills, MCP tools, artifact contracts, and tests.
- Any bug report must include concrete evidence and the no-fix sentence.
- No source, manifest, skill, test, generated asset, runtime `.blueprint/`, host-global, remote, branch, PR, or git-history fix is applied.
