# Phase 4 - Plan 01

## Goal

Audit Blueprint roadmap mutation commands for the strict safety posture captured in D-01 through D-04: preview parity, two-gate destructive scrutiny, partial-failure recovery, and authoritative MCP-returned routing.

## Scope

- Compare roadmap mutation command specs against their TOML manifests, roadmap-admin skill text, and runtime references.
- Inspect `blueprintRoadmapAddPhase`, `blueprintRoadmapInsertPhase`, `blueprintRoadmapRemovePhase`, and `blueprintRoadmapPromoteBacklog` in `src/mcp/tools/phase.ts`.
- Run focused roadmap metadata/tool tests or document the environment blocker.
- Create bug reports only for confirmed or likely defects.

## Tasks

### Task 1: Compare roadmap mutation contracts to manifests and skill references

#### Read First

- `docs/commands/add-phase.md`
- `docs/commands/insert-phase.md`
- `docs/commands/remove-phase.md`
- `docs/commands/plan-milestone-gaps.md`
- `commands/blu-add-phase.toml`
- `commands/blu-insert-phase.toml`
- `commands/blu-remove-phase.toml`
- `commands/blu-plan-milestone-gaps.toml`
- `skills/blueprint-roadmap-admin/SKILL.md`
- `skills/blueprint-roadmap-admin/references/add-phase-runtime-contract.md`
- `skills/blueprint-roadmap-admin/references/insert-phase-runtime-contract.md`
- `.planning/phases/04-roadmap-capture-lightweight-audit/04-CONTEXT.md`

#### Action

- Build a checklist with these exact expected roadmap mutation requirements:
  - `/blu-add-phase` reads the roadmap, previews the next integer phase, confirms, calls `blueprint_roadmap_add_phase` with `expectedPhaseNumber`, scaffolds the returned context path, and routes to `/blu-discuss-phase <phase>`.
  - `/blu-insert-phase` accepts only an integer anchor, previews the next decimal phase, confirms, calls `blueprint_roadmap_insert_phase`, scaffolds the returned context path, records roadmap evolution notes, and routes from returned phase metadata.
  - `/blu-remove-phase` reads the roadmap and target phase, previews deletion and renumbering, requires a second destructive confirmation before `force: true` when execution evidence exists, calls `blueprint_roadmap_remove_phase`, and routes to `/blu-progress`.
  - `/blu-plan-milestone-gaps` reads a saved milestone audit, groups actionable gaps, confirms, appends phases via repeated `blueprint_roadmap_add_phase` calls, and routes to `/blu-discuss-phase <first new phase>`.
  - All four flows keep follow-up guidance inside implemented commands only.
- Compare docs, TOML prompts, skill text, and runtime references for material drift.
- File a bug if drift could cause preview/apply mismatch, skipped destructive confirmation, hand-authored roadmap mutation, or misleading follow-up routing.

#### Acceptance Criteria

- `rg -n "expectedPhaseNumber|integer anchor|force: true|second explicit destructive confirmation|blueprint_roadmap_add_phase|blueprint_roadmap_insert_phase|blueprint_roadmap_remove_phase|/blu-discuss-phase|/blu-progress" docs/commands/add-phase.md docs/commands/insert-phase.md docs/commands/remove-phase.md docs/commands/plan-milestone-gaps.md commands/blu-add-phase.toml commands/blu-insert-phase.toml commands/blu-remove-phase.toml commands/blu-plan-milestone-gaps.toml skills/blueprint-roadmap-admin/SKILL.md skills/blueprint-roadmap-admin/references/*.md` finds expected anchors.
- Any new bug report cites both expected and actual sources in `## Evidence`.

- Verify returned `phaseNumber`, `phasePrefix`, `phaseDir`, `removedPhase`, `renumberedPhases`, `promotedItems`, and `createdPhaseDirs` are specific enough for command prompts to avoid prose-derived guesses.
