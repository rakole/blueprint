# Phase 7 Plan 01: Host Manifests And Runtime Metadata

## Goal

Audit Gemini and Tabnine extension manifests plus shared host metadata for extensionPath-rooted MCP launch, host-specific context files, environment variables, install metadata assumptions, and tracked bundle readiness.

## Scope

In scope:
- `gemini-extension.json`, `tabnine-extension.json`, `GEMINI.md`, and `TABNINE.md`.
- `scripts/lib/extension-hosts.mjs` and `tests/helpers/extension-hosts.ts`.
- `tests/extension-runtime-contracts.test.ts` host manifest, context, and tracked bundle checks.
- Docs that lock host constraints and install model.

Out of scope:
- Changing host manifests, context files, scripts, tests, or generated assets.
- Running real host installs; Plan 04 handles install and smoke readiness.
- Applying any fix to package, source, manifest, test, or generated files.

## Tasks

### Task 1: Compare host manifest contracts to locked docs

#### Read First

- `.planning/phases/07-host-packaging-build-hooks-audit/07-RESEARCH.md`
- `docs/DECISIONS.md`
- `docs/ARCHITECTURE.md`
- `docs/GEMINI-CONSTRAINTS.md`
- `docs/RUNTIME-REFERENCE.md`
- `gemini-extension.json`
- `tabnine-extension.json`
- `GEMINI.md`
- `TABNINE.md`

#### Action

Verify these concrete target states:

- `gemini-extension.json` has `"name": "blueprint"`, `"contextFileName": "GEMINI.md"`, MCP server `blueprint.command` set to `"node"`, first arg exactly `"${extensionPath}/dist/mcp/server.js"`, `BLUEPRINT_HOST` exactly `"gemini"`, and `BLUEPRINT_EXTENSION_PATH` exactly `"${extensionPath}"`.
- `tabnine-extension.json` has `"name": "blueprint"`, `"contextFileName": "TABNINE.md"`, MCP server `blueprint.command` set to `"node"`, first arg exactly `"${extensionPath}/dist/mcp/server.js"`, `BLUEPRINT_HOST` exactly `"tabnine"`, and `BLUEPRINT_EXTENSION_PATH` exactly `"${extensionPath}"`.
- `TABNINE.md` resolves to the same runtime operator guide as `GEMINI.md` and the guide does not expose stale internal checkpoint or drift-repair status as host runtime instructions.
- Host docs state that Blueprint is installed through host extension install/link behavior and that commands must not self-mutate the installed extension directory.

#### Acceptance Criteria

- `rg -n "\"contextFileName\"|\"\\$\\{extensionPath\\}/dist/mcp/server\\.js\"|\"BLUEPRINT_HOST\"|\"BLUEPRINT_EXTENSION_PATH\"" gemini-extension.json tabnine-extension.json` finds all required manifest fields.
- `test "$(readlink TABNINE.md)" = "GEMINI.md"` passes or the exact non-symlink evidence is recorded.
- Any manifest/docs drift is either ruled out with citations or recorded as a Phase 7 bug candidate.

### Task 2: Audit shared extension-host metadata and runtime contract tests

#### Read First

- `scripts/lib/extension-hosts.mjs`
- `tests/helpers/extension-hosts.ts`
- `tests/extension-runtime-contracts.test.ts`
- `package.json`

#### Action

Audit whether shared host metadata and tests cover these target states:

- `KNOWN_EXTENSION_HOSTS` contains exactly `gemini` and `tabnine` host ids with correct binary names, manifest files, context files, host home directories, global Blueprint roots, and install metadata filename expectations.
- `discoverExtensionHosts()` discovers a host when its manifest or context file is present.
- `tests/extension-runtime-contracts.test.ts` asserts manifest entrypoints, host env, host runtime context reuse, tracked built assets, implemented skill metadata, and path-free direct command manifests.
- The test imports runtime source where appropriate and checks the same live command catalog and tool names the package ships.

Run or document:

```bash
npx tsx --test tests/extension-runtime-contracts.test.ts
```

#### Acceptance Criteria

- The targeted runtime-contract command exits 0, or the exact failure/environment blocker is recorded.
- Any unsupported host metadata, stale install metadata filename, missing manifest check, or runtime-contract blind spot that can break install readiness is documented as a bug candidate with exact evidence.

- Run `git status --short`.

## Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| T-01 | A manifest defect makes installed Blueprint fail before MCP startup. | Check manifest paths, env, host context files, and runtime-contract tests together. |
| T-02 | Host metadata drift makes tests or smoke scripts validate a different contract than the manifests ship. | Compare shared script metadata, test helper metadata, and host manifests in the same plan. |
| T-03 | The audit accidentally fixes manifests. | Write only bug docs and summaries; leave source manifests unchanged. |

## Must Haves

- Host manifests must launch `${extensionPath}/dist/mcp/server.js` with the correct host env values.
- Gemini and Tabnine context files must be host-appropriate and traceable to the shared runtime operator guide.
- Shared host metadata must not invent unsupported hosts or stale global state roots.
- Any bug report must include concrete evidence and the no-fix sentence.
- No source, manifest, skill, test, generated asset, runtime `.blueprint/`, installed-extension, host-global, branch, PR, remote, or git-history fix is applied.
