# Phase 7 Plan 03: Advisory Hooks

## Goal

Audit Blueprint's advisory hooks for configuration correctness, stdin/stdout stability, shared security detector reuse, path handling, transcript parsing, built-script parity, and strict advisory-only behavior.

## Scope

In scope:
- `hooks/hooks.json`.
- `src/hooks/shared.ts`, `src/hooks/run-hook.ts`, `src/hooks/read-before-edit.ts`, `src/hooks/blueprint-write-guard.ts`, and `src/hooks/workflow-advisory.ts`.
- `src/shared/security.ts` prompt-boundary and path helpers used by hooks.
- Hook policy docs and hook/security tests.

Out of scope:
- Making hooks block or enforce writes.
- Changing hook matcher config, hook source, shared security helpers, or tests.
- General MCP persistence enforcement outside the hook advisory boundary.

## Tasks

### Task 1: Compare hook policy and configuration

#### Read First

- `docs/HOOKS-POLICIES.md`
- `docs/GEMINI-CONSTRAINTS.md`
- `hooks/hooks.json`
- `tests/hooks.test.ts`
- `tests/built-assets-smoke.test.ts`

#### Action

Verify these concrete target states:

- `hooks/hooks.json` has one `BeforeTool` group with matcher exactly `write_file|replace`.
- The group contains exactly `blueprint-read-before-edit`, `blueprint-blueprint-write-guard`, and `blueprint-workflow-advisory`.
- Every hook has `"type": "command"` and a command matching `node ${extensionPath}/dist/hooks/<name>.js`.
- Hook docs state hooks are advisory only, repo config does not control hook activation, and hooks do not own persistence.
- Built-asset smoke resolves hook commands from `hooks/hooks.json` and executes the built files, not only TypeScript source.

#### Acceptance Criteria

- `rg -n "BeforeTool|write_file\\|replace|blueprint-read-before-edit|blueprint-blueprint-write-guard|blueprint-workflow-advisory|\\$\\{extensionPath\\}/dist/hooks" hooks/hooks.json tests/hooks.test.ts tests/built-assets-smoke.test.ts` finds aligned hook config evidence.
- Any hook config/docs drift that can break host execution or widen hook authority is documented as a bug candidate.

### Task 2: Audit hook behavior and shared security reuse

#### Read First

- `src/hooks/shared.ts`
- `src/hooks/run-hook.ts`
- `src/hooks/read-before-edit.ts`
- `src/hooks/blueprint-write-guard.ts`
- `src/hooks/workflow-advisory.ts`
- `src/shared/security.ts`
- `tests/hooks.test.ts`
- `tests/security-hardening.test.ts`
- `tests/security-docs.test.ts`

#### Action

Audit whether implementation and tests satisfy:

- `readHookInput()` accepts empty stdin as `{}`, uses `safeJsonParseObject` with a 512 KiB max input limit, and does not throw unhandled parse failures through hook output.
- `runHook()` writes JSON to stdout and returns `{}` on unexpected handler failures while logging stderr diagnostics.
- `read-before-edit` ignores non-write tools, new files, and `.blueprint` writes; it warns only on existing non-`.blueprint` targets that were not read via transcript `read_file` or `read_many_files`.
- `.blueprint` write guard ignores non-`.blueprint` writes, warns on prompt-boundary findings, warns on research-like paths missing headings/evidence markers, and remains advisory.
- `workflow-advisory` warns on direct non-`.blueprint` repo writes and stays silent for `.blueprint` paths.
- All warnings return `decision: "allow"` or `{}`; no hook returns `deny`, `block`, or owns state mutation.

Run or document:

```bash
npx tsx --test tests/hooks.test.ts tests/security-hardening.test.ts tests/security-docs.test.ts
```

#### Acceptance Criteria

- The targeted hook/security command exits 0, or the exact failure/environment blocker is recorded.
- Any hook path containment, transcript parsing, prompt-boundary, advisory-only, or test coverage defect is documented as a bug candidate with exact source/test evidence.

## Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| T-01 | Hooks accidentally block host writes or become a hidden permission layer. | Assert advisory-only outputs and compare to hook policy docs. |
| T-02 | Hook path parsing or transcript parsing misses real warnings. | Inspect shared path helpers, transcript traversal, and tests. |
| T-03 | Hook warnings drift from shared security detectors. | Verify `.blueprint` write guard uses `analyzePromptBoundaryText` and security tests cover detector docs. |

## Must Haves

- Hook config must stay rooted at `${extensionPath}/dist/hooks/*.js`.
- Hooks must stay advisory and never own persistence or enforcement.
- `.blueprint` prompt-boundary warnings must reuse shared security detectors.
- Any bug report must include concrete evidence and the no-fix sentence.
- No source, manifest, skill, test, generated asset, runtime `.blueprint/`, installed-extension, host-global, branch, PR, remote, or git-history fix is applied.
