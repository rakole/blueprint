# Phase 7 Plan 04: Install And Clean-Home Smoke Readiness

## Goal

Audit Blueprint's install and smoke readiness for staged shipped bundle contents, host validate/link/install flows, clean-home behavior, install metadata checks, optional Tabnine and live Gemini gates, and disposable probe cleanup.

## Scope

In scope:
- `scripts/gemini-clean-home-smoke.mjs` and its fake-CLI tests.
- `tests/extension-install.integration.ts` containerized host install/link smoke.
- Shared extension host metadata and package script wiring.
- Required and forbidden shipped paths, install metadata, MCP import, hook target containment, clean-home env, and skip behavior.

Out of scope:
- Mutating real user host homes or installed extension directories.
- Creating PRs, pushing branches, or changing package/install scripts.
- Treating missing Docker, Tabnine CLI installer config, or Gemini API key as source defects without supporting packaging evidence.

## Tasks

### Task 1: Audit clean-home smoke script and fake-CLI tests

#### Read First

- `scripts/gemini-clean-home-smoke.mjs`
- `scripts/lib/extension-hosts.mjs`
- `tests/gemini-clean-home-smoke.test.ts`
- `package.json`

#### Action

Verify these concrete target states:

- `scripts/gemini-clean-home-smoke.mjs` accepts `--host gemini|tabnine`, `--repo`, `--home`, `--cli`, and `--keep-home`.
- The smoke requires the host manifest and `dist/mcp/server.js` before running.
- It prepares a clean host home with `HOME`, `USERPROFILE`, `XDG_CONFIG_HOME`, `XDG_CACHE_HOME`, and `XDG_STATE_HOME`.
- It runs `<host> extensions validate <repo> --debug`, `<host> extensions link <repo> --consent`, and `<host> extensions list --output-format json`.
- It verifies the listed extension has `name: "blueprint"` and `path` equal to the repo root.
- It removes auto-created clean homes only after success and preserves them on failure.
- Fake-CLI tests prove ordering, clean-home environment reuse, success output, and failure preservation for both hosts.

Run or document:

```bash
npx tsx --test tests/gemini-clean-home-smoke.test.ts
```

#### Acceptance Criteria

- The clean-home smoke test command exits 0, or the exact failure/environment blocker is recorded.
- Any missing preflight, unsafe cleanup, host-specific path mistake, or list-output parsing defect is documented as a bug candidate.

### Task 2: Audit containerized extension install integration

#### Read First

- `tests/extension-install.integration.ts`
- `tests/helpers/extension-hosts.ts`
- `gemini-extension.json`
- `tabnine-extension.json`
- `hooks/hooks.json`
- `package.json`

#### Action

Audit whether integration coverage satisfies:

- Staging uses tracked files only from `gemini-extension.json`, `tabnine-extension.json`, host context files, `commands`, `skills`, `agents`, `hooks`, `dist`, and `package.json`.
- Staging excludes `src`, `node_modules`, `.planning`, and `.git`.
- Required installed paths include host manifests, context files, `commands/blu.toml`, `commands/blu-help.toml`, `skills/blueprint-router/SKILL.md`, `agents/blueprint-planner.md`, `hooks/hooks.json`, `dist/mcp/server.js`, all three built hook scripts, and `package.json`.
- `gemini` can install `@google/gemini-cli`; Tabnine smoke is skipped unless `BLUEPRINT_TABNINE_CLI_INSTALL_COMMAND` is set.
- Link and install modes validate extension enablement, debug list visibility, install metadata type, forbidden path exclusion, MCP server import, and hook command target containment.
- Live Gemini prompt smoke is skipped unless `GEMINI_API_KEY` is present.

Run when environment supports Docker and host prerequisites:

```bash
npm run test:integration:extension
```

If unavailable, record the exact blocker, such as Docker unavailable, missing `BLUEPRINT_TABNINE_CLI_INSTALL_COMMAND`, network install failure, or missing `GEMINI_API_KEY`.

#### Acceptance Criteria

- The integration command exits 0 or records precise skips/blockers with enough detail for a later verifier.
- Any staged bundle omission, forbidden path leak, metadata mismatch, hook/MCP escape, or host validation defect is documented as a bug candidate.
- Optional host/auth skips are not reported as Blueprint defects unless the test or manifest handles them incorrectly.

## Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| T-01 | Smoke probes mutate real host homes or installed extensions. | Use fake CLI tests, clean temp homes, and containers; never target the user's actual host home. |
| T-02 | A shipped bundle passes local tests but omits files needed after install. | Check staged required/forbidden path assertions and installed MCP/hook import checks. |
| T-03 | Optional host/auth prerequisites create false defects. | Separate Docker, Tabnine installer, network, and Gemini API blockers from Blueprint packaging failures. |

## Must Haves

- Clean-home smoke behavior must validate, link, list, verify, and clean up safely.
- Containerized integration must verify staged bundle contents, link/install modes, metadata, MCP import, and hook command containment.
- Optional host/auth skips must be classified honestly.
- Any bug report must include concrete evidence and the no-fix sentence.
- No source, manifest, skill, test, generated asset, runtime `.blueprint/`, installed-extension, host-global, branch, PR, remote, or git-history fix is applied.
