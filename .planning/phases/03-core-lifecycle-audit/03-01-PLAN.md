# Phase 3 - Plan 01

## Goal

Audit Blueprint's pre-planning discovery commands in the user-facing order so context, research, UI-spec or skip-rationale artifacts, shared checkpoint ownership, and state handoffs remain deterministic and MCP-owned.

## Scope

- Compare `/blu-discuss-phase`, `/blu-research-phase`, and `/blu-ui-phase` command specs against their TOML manifests.
- Trace each command into `skills/blueprint-phase-discovery/SKILL.md`, its command-specific runtime reference, and relevant MCP tools.
- Inspect artifact contract handling for `phase.context`, `phase.discussion-log`, `phase.research`, and `phase.ui-spec`.
- Inspect shared checkpoint ownership and mode guards in `src/mcp/tools/phase.ts`.
- Run targeted discovery tests or document the environment blocker.
- Create bug reports only for confirmed or likely defects.

## Tasks

### Task 1: Compare discovery command contracts to manifests and skill references

#### Read First

- `docs/commands/discuss-phase.md`
- `docs/commands/research-phase.md`
- `docs/commands/ui-phase.md`
- `commands/blu-discuss-phase.toml`
- `commands/blu-research-phase.toml`
- `commands/blu-ui-phase.toml`
- `skills/blueprint-phase-discovery/SKILL.md`
- `skills/blueprint-phase-discovery/references/discuss-phase-runtime-contract.md`
- `skills/blueprint-phase-discovery/references/research-phase-runtime-contract.md`
- `skills/blueprint-phase-discovery/references/ui-phase-runtime-contract.md`
- `.planning/phases/03-core-lifecycle-audit/03-CONTEXT.md`

#### Action

- Build a checklist with these exact expected discovery requirements:
  - `/blu-discuss-phase` reads phase context, plan inventory, context and discussion-log contracts, existing artifacts, and shared checkpoint state before write decisions.
  - `/blu-discuss-phase` deletes checkpoints only with `expectedOwnerCommand: "/blu-discuss-phase"` and `expectedMode: "discuss"`.
  - `/blu-research-phase` stops on missing `XX-CONTEXT.md`, honors `research.external_sources`, reads `phase.research`, and deletes only research-owned checkpoints.
  - `/blu-ui-phase` uses `XX-UI-SPEC.md` as the single durable artifact for either a real UI contract or explicit skip rationale.
  - All three flows persist substantive final artifacts through `blueprint_phase_artifact_write`, not raw file edits or scaffold placeholders.
  - All follow-up routing stays inside implemented commands.
- Compare docs, TOML prompts, skill text, and runtime references for material drift.
- If a mismatch creates user-impact or lifecycle-state risk, create the next `docs/bugs/BPBUG-###-*.md` report using `docs/bugs/TEMPLATE.md`.

#### Acceptance Criteria

- `rg -n "expectedOwnerCommand|expectedMode|phase\\.research|phase\\.ui-spec|blueprint_phase_artifact_write|blueprint_state_update" docs/commands/discuss-phase.md docs/commands/research-phase.md docs/commands/ui-phase.md commands/blu-discuss-phase.toml commands/blu-research-phase.toml commands/blu-ui-phase.toml skills/blueprint-phase-discovery/SKILL.md skills/blueprint-phase-discovery/references/*.md` finds the required contract anchors.
- Any new bug report cites both expected and actual sources in `## Evidence`.

### Task 2: Audit discovery MCP tool and artifact-contract substrate

#### Read First

- `src/mcp/tools/phase.ts`
- `src/mcp/artifact-contracts/index.ts`
- `src/mcp/artifact-contracts/schemas/phase.plan.model.schema.json`
- `tests/phase-discovery-tools.test.ts`
- `tests/phase-discovery-discuss.test.ts`
- `tests/phase-discovery-research.test.ts`
- `tests/phase-discovery-ui.test.ts`

#### Action

- Inspect `blueprintPhaseArtifactRead`, `blueprintPhaseArtifactWrite`, `blueprintPhaseResearchStatus`, `blueprintPhaseCheckpointGet`, `blueprintPhaseCheckpointPut`, and `blueprintPhaseCheckpointDelete`.
- Verify checkpoint persistence requires a structured checkpoint object with `ownerCommand`, `completedAreas`, `remainingAreas`, `decisions`, `deferredIdeas`, `canonicalReferences`, and `resumeMeta`.
- Verify checkpoint reads and deletes expose or enforce owner/mode safety, including legacy warning behavior.
- Verify artifact writes validate substantive discovery content and do not treat scaffold placeholders as completed artifacts.
- Run targeted tests:
  - `npx tsx --test tests/phase-discovery-discuss.test.ts tests/phase-discovery-research.test.ts tests/phase-discovery-ui.test.ts tests/phase-discovery-tools.test.ts`
- If tests fail, classify whether the failure is defect evidence, an environment blocker, or unrelated pre-existing breakage.

#### Acceptance Criteria

- `rg -n "ownerCommand|resumeMeta|expectedOwnerCommand|expectedMode|Refusing to delete|blueprintPhaseCheckpoint" src/mcp/tools/phase.ts tests/phase-discovery-tools.test.ts tests/phase-discovery-discuss.test.ts tests/phase-discovery-research.test.ts` finds checkpoint evidence.
- The summary records targeted test pass/fail output or a concrete environment blocker.

### Task 3: Record discovery-flow findings

#### Read First

- `docs/bugs/TEMPLATE.md`
- `docs/bugs/INDEX.md`
- `.planning/phases/03-core-lifecycle-audit/03-PATTERNS.md`

#### Action

- For every confirmed or likely discovery-flow defect from Tasks 1 and 2:
  - Create one `docs/bugs/BPBUG-###-short-slug.md` file.
  - Set `discovery_phase: 3`.
  - Use `surface: command`, `surface: skill`, `surface: MCP tool`, `surface: docs`, or `surface: tests` as appropriate.
  - Include exact file paths, line-level references where useful, and command/test evidence.
  - Include the exact no-fix sentence from the template.
  - Add or update the matching row in `docs/bugs/INDEX.md`.
- If no defect is found in this plan, record that result in the Plan 01 summary and leave final Phase 3 slice-row reconciliation to Plan 05.

#### Acceptance Criteria

- If a bug was created, `rg -n "discovery_phase: 3|Expected Behavior|Actual Behavior|Evidence|Verification Steps|No Fix Applied" docs/bugs/BPBUG-*.md` finds required sections.
- `git status --short` shows no source, manifest, skill, test, build, generated asset, `.blueprint/`, installed-extension, or host-global mutation from this audit.
