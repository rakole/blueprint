## Phase Boundary

Repair the phase-planning and future-command documentation so Blueprint does not drift again after Wave 0. This phase keeps `docs/DRIFT.MD` as the canonical repair ledger, reasserts the intended Blueprint shape for retained future commands, and aligns roadmap-era docs with the architecture that was actually chosen: thin Gemini commands, skill-led orchestration, bounded custom agents for heavy work, and MCP-owned persistence. ## Implementation Decisions

### Repair Ledger Ownership
- **D-01:** Reopen `docs/DRIFT.MD` as the live canonical repair ledger for Phase 2.2 instead of treating it as a closed historical note for Phase 2.1 only.

### Execution Shape
- **D-02:** Structure Phase 2.2 as horizontal repair tracks rather than command-by-command delivery.
- **D-03:** The primary track is future-phase documentation and contract repair so later phases do not drift away from the intended Blueprint architecture.
- **D-04:** Wave 0 runtime cleanup remains in scope only as a guardrail and verification layer, not as the main purpose of the phase.

### Exposure Policy During Repair
- **D-05:** Future commands may be documented early with their full intended architecture, including skills and custom agents.
- **D-06:** `/blu`, `/blu:help`, and `/blu:progress` must only surface a command as runnable when the full runtime shape is present, even if the docs already describe the target architecture.

### Parity Bar For Future Phases
- **D-07:** Use behavior-intent parity with GSD for retained future phases: commands stay lean, orchestration lives in skills, heavy work lives in bounded custom agents, and persistence stays in MCP tools.
- **D-08:** Blueprint does not need file-for-file parity with upstream GSD, but future docs must not simplify retained commands into command-heavy flows that remove skills or agents without an explicit written Blueprint reason.
- **D-09:** Differences from GSD must be limited to explicit Blueprint deltas such as `/blu` naming, `.blueprint/` state, Gemini-native routing, advisory hooks, no slash-command chaining, no repo-level hook config, and no self-update.

### the agent's Discretion
- Exact decomposition of future repair work into plans and subtracks, as long as the primary emphasis stays on preventing future documentation drift.
- Exact wording and examples used when rewriting future command specs, provided they preserve the locked command-skill-agent-MCP separation.
- Exact custom-agent roster for future commands when multiple bounded-agent decompositions could satisfy the same behavior-intent parity requirement.

</decisions>
