### Pattern 1: Catalog-Driven Runtime Truth

**What:** Treat `docs/COMMAND-CATALOG.md` as declared intent, then let `blueprint_command_catalog` compute actual routability from manifest presence, skill presence, and required MCP tool availability. [VERIFIED: src/mcp/tools/project.ts, docs/DRIFT.MD]

**When to use:** For every change that touches command status, command exposure, or blocked-vs-implemented behavior. [VERIFIED: AGENTS.md, docs/DRIFT.MD]

**Example:**
```typescript
// Source: src/mcp/tools/project.ts
const missingTools = requiredTools.filter((toolName) => !AVAILABLE_TOOL_NAMES.has(toolName));
const requiredToolsSatisfied = missingTools.length === 0;

let status = parsedRow.declaredStatus;

if (manifestExists && skillExists && requiredToolsSatisfied) {
  status = "implemented";
} else if (manifestExists || skillExists) {
  status = "repairing";
} else if (blockedBy.length > 0) {
  status = "blocked";
}
```

**When to use:** When repairing or authoring any future-command spec in `docs/commands/`. [VERIFIED: docs/COMMAND-CATALOG.md, docs/commands/next.md]

### Anti-Patterns to Avoid

- **Editing only one metadata surface:** Updating `docs/COMMAND-CATALOG.md` without also checking `docs/commands/*.md`, `docs/SKILLS-AND-AGENTS.md`, and `docs/GSD-RUNTIME-MIGRATION.md` reopens exactly the drift this phase exists to stop. [VERIFIED: tests/command-contract-docs.test.ts, docs/DRIFT.MD]
- **Using declared status as exposure truth:** `project.ts` recomputes `implemented` from substrate presence, so changing the table cell alone is not a safe or sufficient runtime change. [VERIFIED: src/mcp/tools/project.ts]
- **Flattening high-risk commands into command-only prose:** Planned commands such as `quick` and `code-review-fix` already name bounded agents and MCP-owned writes explicitly; removing that structure would violate DRIFT-05. [VERIFIED: docs/commands/quick.md, docs/commands/code-review-fix.md, .planning/REQUIREMENTS.md]
- **Changing `blueprint_command_catalog` semantics during this phase:** Repo guardrails explicitly forbid changing status semantics in Phase 2.2. [VERIFIED: AGENTS.md, docs/DRIFT.MD]

## Common Pitfalls

### Pitfall 1: Single-Surface Metadata Repair

**What goes wrong:** A command’s skill ownership or wave metadata is updated in one file but not the others. [VERIFIED: docs/DRIFT.MD, tests/command-contract-docs.test.ts]

**Why it happens:** The command contract is spread across the catalog, command spec, skills inventory, and migration matrix. [VERIFIED: docs/COMMAND-CATALOG.md, docs/SKILLS-AND-AGENTS.md, docs/GSD-RUNTIME-MIGRATION.md]

**How to avoid:** Make the plan explicitly sequence canonical edits first, then run the doc-contract test immediately after each repair cluster. [VERIFIED: tests/command-contract-docs.test.ts]

**Warning signs:** `Skill inventory mismatch`, `Migration matrix mismatch`, or missing command spec assertions begin failing. [VERIFIED: tests/command-contract-docs.test.ts]

### Pitfall 2: Confusing Declared Status With Runtime Status

**What goes wrong:** A row in `docs/COMMAND-CATALOG.md` is changed and reviewers assume the command has become runnable. [VERIFIED: docs/COMMAND-CATALOG.md, src/mcp/tools/project.ts]

**Why it happens:** The catalog carries both declared intent and derived runtime state. [VERIFIED: src/mcp/tools/project.ts]

**How to avoid:** Keep Phase 02.2 changes away from manifest creation and MCP-tool additions unless the phase scope explicitly says otherwise. [VERIFIED: AGENTS.md, docs/DRIFT.MD]

**Warning signs:** `declaredStatus` changes while `blockedBy` remains populated or `implemented` stays `false`. [VERIFIED: src/mcp/tools/project.ts, tests/command-catalog.test.ts]

### Pitfall 3: PATH Assumptions In Verification

**What goes wrong:** A quick test command fails because `tsx` is not globally installed. [VERIFIED: `tsx --test tests/command-catalog.test.ts tests/command-contract-docs.test.ts tests/drift-repair-docs.test.ts`]

**Why it happens:** The repo depends on local devDependencies, not a global CLI toolchain. [VERIFIED: package.json, `node_modules/.bin/tsx --version`]

**How to avoid:** Use `node_modules/.bin/tsx --test ...` for the fast loop and `npm test` for the full suite. [VERIFIED: `node_modules/.bin/tsx --test tests/command-catalog.test.ts tests/command-contract-docs.test.ts tests/drift-repair-docs.test.ts`, npm test]

**Warning signs:** `zsh: command not found: tsx`. [VERIFIED: `tsx --test tests/command-catalog.test.ts tests/command-contract-docs.test.ts tests/drift-repair-docs.test.ts`]

### Pitfall 4: Closing The Phase In Docs Before Planning Artifacts Catch Up

**What goes wrong:** The drift ledger reads as satisfied while roadmap requirements and phase plans still show the phase as active and pending. [VERIFIED: docs/DRIFT.MD, .planning/ROADMAP.md, .planning/REQUIREMENTS.md]

**Why it happens:** The repo already contains the repaired docs/tests, but the planning flow has not yet produced the phase plans that formally execute and close Phase 02.2. [VERIFIED: docs/DRIFT.MD, .planning/ROADMAP.md, .planning/phases/02.2-urgent-drift-repair-follow-up/02.2-CONTEXT.md]

**How to avoid:** Start the plan with a ratification track that preserves the current repaired baseline and defines the formal closeout criteria for the planning system. [VERIFIED: .planning/ROADMAP.md, .planning/STATE.md, docs/DRIFT.MD]

**Warning signs:** `docs/DRIFT.MD` shows all exit criteria checked while `.planning/ROADMAP.md` still shows `0/4` plans and `.planning/REQUIREMENTS.md` still shows `DRIFT-05` through `DRIFT-07` as pending. [VERIFIED: docs/DRIFT.MD, .planning/ROADMAP.md, .planning/REQUIREMENTS.md]
