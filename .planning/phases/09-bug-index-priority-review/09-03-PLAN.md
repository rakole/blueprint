# Phase 9 Plan 03: Repair Priority, Batches, And Open Questions

## Goal

Turn the reconciled bug inventory into a repair-planning view with priority bands, repair batches, and a separate verification-question lane without selecting or applying fixes.

## Scope

In scope:
- `Now`, `Next`, and `Later` repair-priority bands.
- Repair batches grouped by shared repair direction and validation surface.
- Repaired/history items separated from active candidates.
- Verification questions or explicit no-open-question outcome.
- `09-REPAIR-QUEUE.md` and `09-03-SUMMARY.md`.

Out of scope:
- Implementing repairs.
- Creating GitHub issues, projects, or milestones.
- Running broad `npm test` unless execution chooses to confirm a status claim; grep/status evidence is sufficient for planned Phase 9 outputs.

## Tasks

#### Files

- `docs/bugs/INDEX.md`
- `.planning/phases/09-bug-index-priority-review/09-REPAIR-QUEUE.md`

- `.planning/phases/09-bug-index-priority-review/09-02-SUMMARY.md`

- `docs/bugs/BPBUG-*.md`

#### Action

Create `09-REPAIR-QUEUE.md` and add or update a matching section in `docs/bugs/INDEX.md` named `## Repair Priority`.

Use exactly these bands:

- `Now`
- `Next`
- `Later`

For each active bug candidate, include:

- `Bug ID`
- `Priority Band`
- `Why This Band`
- `Repair Leverage`
- `Validation Surface`

Use D-01 through D-04:

- Do not rank by severity alone.
- Separate repaired/history items before ranking.
- Break ties by cross-cutting safety leverage or recurrence risk.
- Avoid precise numeric scoring.

If Plan 01 confirmed BPBUG-004 as repaired/verified, keep it in a `Repaired / History` subsection and do not assign it to `Now`, `Next`, or `Later`.
