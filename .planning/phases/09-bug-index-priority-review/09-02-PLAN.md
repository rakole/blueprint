# Phase 9 Plan 02: Duplicate And Related-Bug Review

## Goal

Apply the Phase 9 duplicate policy across BPBUG-001 through BPBUG-005 so the inventory clearly distinguishes true duplicates from related-but-distinct bugs and preserves canonical repair evidence.

## Scope

In scope:
- Duplicate policy text in `docs/bugs/INDEX.md`.
- Related-bug and root-cause cluster links for real BPBUG reports.
- Explicit no-duplicate outcome if no same-defect/same-repair-path duplicates are found.
- `09-02-SUMMARY.md`.

Out of scope:
- Repair priority ordering; Plan 03 owns it.
- Creating new root-cause clusters unless execution finds an obvious missing link.
- Fixing source, docs/runtime drift, tests, generated assets, or command behavior.

## Tasks

### Task 1: Compare duplicate candidates against policy

#### Files

- `.planning/phases/09-bug-index-priority-review/09-02-SUMMARY.md`

#### Read First

- `.planning/phases/09-bug-index-priority-review/09-INVENTORY-REVIEW.md`
- `docs/bugs/INDEX.md`
- `docs/bugs/BPBUG-001-ship-undo-report-contracts-underconstrained.md`
- `docs/bugs/BPBUG-002-cleanup-lacks-behavioral-regression-coverage.md`
- `docs/bugs/BPBUG-003-mcp-tools-docs-stale-update-return-shapes.md`
- `docs/bugs/BPBUG-004-stale-built-bundle-omits-audit-fix-generated-assets.md`
- `docs/bugs/BPBUG-005-repo-root-guard-accepts-any-git-entry.md`

#### Action

For each pair or obvious cluster among BPBUG-001 through BPBUG-005, compare:

- user-visible defect
- affected surface
- current status
- evidence source
- likely cause
- suggested fix direction

Mark `duplicate` only if both the user-visible defect and repair path are the same. Otherwise keep the reports separate and record the relationship as root-cause or related-bug context.

#### Verify

```xml
<verify>
  <automated>rg -n "Duplicate Review|same user-visible defect|same repair path|No duplicate reports were marked|canonical" .planning/phases/09-bug-index-priority-review/09-02-SUMMARY.md</automated>
</verify>
```

#### Acceptance Criteria

- `09-02-SUMMARY.md` records the duplicate decision for BPBUG-001 through BPBUG-005.
- No report is marked duplicate unless the summary names its canonical report and same repair path.
- Related-but-distinct bugs remain separate.

#### Action

Update `docs/bugs/INDEX.md`:

- Keep or improve `## Duplicate And Related Findings`.
- State that duplicate status requires the same user-visible defect and same repair path.
- Add a concise duplicate review outcome: either `No duplicate reports are currently marked` or a table with `Duplicate`, `Canonical`, `Reason`, and `Repair Path`.

Update `## Related Bugs` sections only where needed:
