# Phase 2 - Plan 01

## Goal

Audit the foundational read-only router outputs and readiness branches so user-facing `/blu`, `/blu-help`, `/blu-progress`, and `/blu-next` guidance stays limited to implemented commands and routes map-first states correctly.

## Scope

- Compare command specs and manifests for `/blu`, `/blu-help`, `/blu-progress`, and `/blu-next`.
- Compare runtime status/next-action logic in `src/mcp/tools/project.ts` and `src/mcp/tools/state.ts`.
- Inspect tests that encode greenfield, brownfield, `mapping-incomplete`, `mapped-only`, `partial`, initialized, waiting-state, and implemented-only behavior.
- Create bug reports only for confirmed or likely defects.
- Update `docs/bugs/INDEX.md` Phase 2 slice coverage with examined router surfaces and bug ids or no-defect evidence.

## Tasks

### Task 1: Compare user-facing router contracts to manifests

#### Read First

- `docs/commands/root-router.md`
- `docs/commands/help.md`
- `docs/commands/progress.md`
- `docs/commands/next.md`
- `commands/blu.toml`
- `commands/blu-help.toml`
- `commands/blu-progress.toml`
- `commands/blu-next.toml`
- `.planning/phases/02-bootstrap-router-config-audit/02-CONTEXT.md`

#### Action

- Build a checklist with these exact expected router requirements:
  - `/blu`, `/blu-help`, and `/blu-progress` recommend or route only commands whose catalog entry is `implemented: true`.
  - `/blu-next` never recommends commands whose catalog entry is not implemented.
  - Uninitialized brownfield and `mapping-incomplete` states route to `/blu-map-codebase`.
  - Greenfield/scaffold-only uninitialized and `mapped-only` states route to `/blu-new-project`.
  - Partial `.blueprint/` state routes to `/blu-health`.
  - Waiting state is named explicitly as missing artifact, partial repo repair, verification debt, blocked substrate, or mapping prerequisite.
- Use `rg` or file reads to verify each expected phrase appears in the corresponding docs and manifests.
- If a mismatch is material and user-facing, create the next `docs/bugs/BPBUG-###-*.md` report using `docs/bugs/TEMPLATE.md`.

#### Acceptance Criteria

- `rg -n "implemented: true|/blu-map-codebase|/blu-new-project|/blu-health|waiting state|mapping prerequisite" commands/blu.toml commands/blu-help.toml commands/blu-progress.toml commands/blu-next.toml docs/commands/root-router.md docs/commands/help.md docs/commands/progress.md docs/commands/next.md` finds matches.
- Any new bug report cites both sides of the mismatch in `## Evidence`.

### Task 2: Audit runtime readiness and next-action branches

#### Read First

- `src/mcp/tools/project.ts`
- `src/mcp/tools/state.ts`
- `tests/help-progress-health.test.ts`
- `tests/next.test.ts`

#### Action

- Inspect `blueprintProjectStatus()` and `deriveNextAction()` for these exact branch outcomes:
  - `uninitialized` plus brownfield unmapped -> `/blu-map-codebase` before `/blu-new-project`.
  - `mapping-incomplete` -> `/blu-map-codebase`.
  - `mapped-only` -> `/blu-new-project`.
  - `partial` -> `/blu-health`.
  - initialized phase states choose implemented discovery/lifecycle commands only.
- Inspect tests for coverage of these branches and note any missing or contradictory coverage.
- Run targeted tests if dependencies are installed:
  - `npx tsx --test tests/help-progress-health.test.ts tests/next.test.ts`
- If the test command cannot run because dependencies are missing, record the environment blocker in the plan summary and use static file evidence only.

#### Acceptance Criteria

- `rg -n "mapping-incomplete|mapped-only|brownfield|/blu-map-codebase|/blu-new-project|/blu-health" src/mcp/tools/project.ts src/mcp/tools/state.ts tests/help-progress-health.test.ts tests/next.test.ts` finds branch and test evidence.
- If tests are run, the summary records pass/fail output and any failure used as bug evidence.

### Task 3: Record router audit findings

#### Read First

- `docs/bugs/TEMPLATE.md`
- `docs/bugs/INDEX.md`
- `.planning/phases/02-bootstrap-router-config-audit/02-PATTERNS.md`

#### Action

- For every confirmed or likely router/readiness defect from Tasks 1 and 2:
  - Create one `docs/bugs/BPBUG-###-short-slug.md` file.
  - Set `discovery_phase: 2`.
  - Use `surface: command`, `surface: MCP tool`, `surface: tests`, or `surface: docs` as appropriate.
  - Include exact file paths and command/test evidence.
  - Include the exact no-fix sentence from the template.
  - Add or update the matching row in `docs/bugs/INDEX.md`.
- If no defect is found in this plan, update only the Phase 2 slice coverage row in `docs/bugs/INDEX.md` with `router/readiness surfaces examined; none found in Plan 01`.

#### Acceptance Criteria

- `rg -n "Phase 2|router|readiness|Plan 01" docs/bugs/INDEX.md` finds the slice coverage update.
- If a bug was created, `rg -n "discovery_phase: 2|Expected Behavior|Actual Behavior|Evidence|Verification Steps|No Fix Applied" docs/bugs/BPBUG-*.md` finds required sections.
