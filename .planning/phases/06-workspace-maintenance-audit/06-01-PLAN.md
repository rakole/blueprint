# Phase 6 Plan 01: Workspace Registry And Teardown

## Goal

Audit Blueprint workspace creation and removal for host-global registry atomicity, workspace manifest consistency, worktree/clone safety, exact target resolution, dirty-tree and drift blockers, and explicit confirmation gates.

## Scope

In scope:
- `/blu-new-workspace` and `/blu-remove-workspace` docs, command manifests, and maintenance skill contracts.
- `blueprint_workspace_registry_get`, `blueprint_workspace_create`, and `blueprint_workspace_remove`.
- Host-global registry path resolution through `src/mcp/runtime-host.ts`.
- Registry lock, temp/backup write, rollback, worktree/clone creation, manifest verification, and teardown behavior.
- Focused workspace metadata and tools tests.

Out of scope:
- Running real workspace creation against the user's actual host-global registry.
- Applying source, manifest, skill, runtime, test, generated asset, workspace, branch, or git-history fixes.
- Workstream project-local state; that is Plan 02.

#### Action

Verify that the workspace contracts require these concrete target states:

- `new-workspace` reads `blueprint_config_get` in effective scope before defaulting the workspace root, then reads `blueprint_workspace_registry_get` before mutation.
- `new-workspace` previews workspace name, resolved workspace path, repo list, strategy, branch, per-repo targets, `.blueprint-workspace.json`, host-global `workspaces.json`, and registry mutation plan before `blueprint_workspace_create`.
- `new-workspace` keeps `dirty-working-tree`, `invalid-workspace-source`, `workspace-conflict`, `new-workspace-confirmation`, and explicit worktree-to-clone fallback approval visible.
- `remove-workspace` resolves the exact host-global registry entry before mutation, blocks on `workspace-not-found`, `workspace-path-ambiguity`, `registry-drift`, `malformed-workspace-registry`, `dirty-working-tree`, and `ask-user-unavailable`, and previews the canonical path, manifest, repo members, strategies, and teardown plan before `blueprint_workspace_remove`.
- Both flows keep host-global state under `~/.<host>/blueprint/workspaces.json`, do not invent `.blueprint/` or `.planning/` workspace registries, and do not present planned-only commands as runnable.

### Task 2: Audit registry atomicity, locks, rollback, and worktree safety

#### Read First

- `src/mcp/runtime-host.ts`
- `src/mcp/tools/workspace.ts`
- `tests/workspace-tools.test.ts`
- `tests/new-workspace-metadata.test.ts`
- `tests/remove-workspace-metadata.test.ts`

#### Action

Audit whether the implementation and tests satisfy D-01 through D-06:

- `resolveBlueprintRuntimeHost()` must keep `workspaceRegistryPath` host-global and configurable only through documented host/global environment behavior.
- `readWorkspaceRegistryDocument` and `writeWorkspaceRegistryDocument` must reject malformed registries, sequence temp/backup writes deterministically, and avoid silent recovery from stale backup files as authoritative primary state.
- `withWorkspaceRegistryLock` must cover create and remove mutations, refresh lease state during long operations, and avoid leaving stale locks that permanently block work.
- `blueprintWorkspaceCreate` must validate source repos, target paths, branch names, installed-extension targets, duplicate registry entries, and worktree/clone member creation before or during mutation with rollback of partial disk state and newly-created branches.
- `blueprintWorkspaceRemove` must verify registry and manifest agreement, clean repo members, worktree registration, exact path matches, and rollback of partial teardown plus registry changes when failure occurs.
- The implementation should not overuse `git worktree remove --force` or hide stale worktree metadata risks; if force is used, determine whether it is bounded by prior cleanliness and exact-member verification.

Run or document the targeted workspace test command:

```bash
npx tsx --test tests/new-workspace-metadata.test.ts tests/remove-workspace-metadata.test.ts tests/workspace-tools.test.ts
```

#### Acceptance Criteria

- The targeted workspace command exits 0, or the exact failure/environment blocker is recorded.
- Any registry/write/lock/rollback path that can leave partial registry entries, remove the wrong workspace, write outside `~/.<host>/blueprint/`, or mutate installed-extension targets is documented as a Phase 6 bug candidate with exact source/test evidence.
- Any missing regression coverage is documented as a test-gap bug only if the uncovered path has concrete maintenance risk and no equivalent test already exists.

## Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| T-01 | The audit accidentally creates or removes a real workspace. | Prefer static reads and existing tests; any behavior probe must use disposable temp roots and `BLUEPRINT_GLOBAL_HOME`. |
| T-02 | Registry rollback defects are missed because happy-path tests pass. | Inspect failure-injection, backup, rollback, and lock lease tests before closing. |
| T-03 | Worktree removal looks safe in prompt text but differs from Git safety expectations. | Compare implementation and tests to Git worktree clean-removal and force behavior before conclusion. |

## Must Haves

- D-01, D-02, D-04, D-05, D-13, D-14, and D-15 are explicitly covered by this plan.
- Workspace registry state must stay host-global under `~/.<host>/blueprint/workspaces.json`.
- Workspace create/remove must be audited through docs, manifests, skills, MCP tools, runtime host helpers, and targeted tests.
- Any bug report must include concrete evidence and the no-fix sentence.
- No source, manifest, skill, test, generated asset, runtime `.blueprint/`, installed-extension, host-global, workspace, branch, PR, remote, or git-history fix is applied.
