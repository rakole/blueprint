# Phase 6 Plan 04: Advisory Update And Patch Replay

## Goal

Audit `/blu-update` and `/blu-reapply-patches` for installed-extension read-only boundaries, host-global update and patch registry persistence, latest-version uncertainty, patch preview/replay parity, compatibility blockers, installed-extension target guards, and replay audit recording.

## Scope

In scope:
- `/blu-update` docs, manifest, skill contract, MCP tools, and update tests.
- `/blu-reapply-patches` docs, manifest, skill contract, patch registry tools, and patch tests.
- Host-global state paths under `~/.<host>/blueprint/updates/` and `~/.<host>/blueprint/patches/`.
- Git patch preview/all-or-nothing expectations as a reference standard.

Out of scope:
- Mutating the installed extension directory.
- Replaying real patches in the user's repo.
- Creating or changing real host-global patch/update registries.
- Fixing source, docs, manifests, skills, tests, or generated assets.

- `tests/update-metadata.test.ts`

#### Action

Verify that `/blu-update` satisfies D-03 and D-10:

- `blueprint_update_check` is always called first and returns host, extension path, installed version, install provenance, latest-version lookup status, latest version, update availability, and warnings.
- Extension path handling stays read-only; no command, skill, or tool path writes into the installed extension directory.
- Latest-version lookup failures, ambiguous provenance, missing extension path, malformed installed metadata, unsupported remote, or network unavailability remain visible as unknown/manual fallback states rather than false precision.
- `blueprint_update_plan` writes only under returned host-global `savedPaths.updatesDir`, `savedPaths.metadataPath`, and `savedPaths.checklistPath`.
- Checklist write failures return manual update steps without pretending files were saved.
- User-facing guidance ends with restart/manual fallback instructions and does not imply an in-session self-update or hot reload.

Run or document the targeted update test command:

```bash
npx tsx --test tests/update-metadata.test.ts tests/update-tools.test.ts
```

#### Acceptance Criteria

- The targeted update command exits 0, or the exact failure/environment blocker is recorded.
- Any update behavior that mutates installed-extension files, writes repo-local update artifacts, overstates latest-version certainty, omits restart guidance, or hides checklist write failure is documented as a Phase 6 bug candidate with exact evidence.

- `tests/patch-tools.test.ts`

#### Action

Verify that `/blu-reapply-patches` satisfies D-03, D-11, and D-12:

- Patch state is read from `~/.<host>/blueprint/patches/` through `blueprint_patch_list`; no `.blueprint/reports/` or repo-local patch registry is invented.
- Replay scope names selected patch ids, tracked files, compatibility notes, registry path, patch paths, audit paths, and target HEAD.
- Dirty tree, malformed registry, missing patch payload, compatibility mismatch, preview conflict, and installed-extension target blockers stop before mutation.
- `blueprint_patch_reapply` with `dryRun: true` is the authoritative preview and conflict check.
- The confirmed replay uses the same selected patch ids as preview and does not widen to all compatible patches unless that was the previewed and confirmed scope.
- The implementation uses `git apply --check`-style preview and default all-or-nothing replay behavior rather than partial `--reject` behavior.
- `blueprint_patch_record` appends replay audit entries under the host-global patch registry after replay or a cleanly handled replay outcome.

Run or document the targeted patch command:

```bash
npx tsx --test tests/reapply-patches-metadata.test.ts tests/patch-tools.test.ts
```

#### Acceptance Criteria

- The targeted patch command exits 0, or the exact failure/environment blocker is recorded.
- Any patch replay behavior that applies a broader patch set than previewed, mutates before clean preview, accepts unsafe installed-extension targets, applies partial patches, writes audit data outside `~/.<host>/blueprint/patches/`, or loses compatibility/conflict visibility is documented as a Phase 6 bug candidate with exact evidence.

## Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| T-01 | The audit mutates installed extension files or real host-global update/patch registries. | Use static reads and existing tests first; any probe must use disposable temp roots and `BLUEPRINT_GLOBAL_HOME`. |
| T-02 | `/blu-update` appears precise when latest-version provenance is unavailable. | Check check-result statuses, warnings, manual fallback, and saved checklist output. |
| T-03 | Patch replay applies a different set than the preview or partially applies a patch. | Compare command/skill contract with `blueprintPatchReapply` dry-run and non-dry-run code plus patch tests. |

## Must Haves

- D-03, D-10, D-11, D-12, D-13, D-14, and D-15 are explicitly covered by this plan.
- `/blu-update` must remain advisory and non-self-mutating.
- `/blu-reapply-patches` must preserve `preflight -> preview -> confirm -> replay -> record`.
- Any bug report must include concrete evidence and the no-fix sentence.
- No source, manifest, skill, test, generated asset, runtime `.blueprint/`, installed-extension, host-global, workspace, patch-registry, branch, PR, remote, or git-history fix is applied.
