# Phase 8 Plan 03: Concern Map Schema Parser Filesystem And Scaling Triage

## Goal

Triage `.planning/codebase/CONCERNS.md` high-risk leads against current Blueprint source, docs, tests, and prior Phase 8 matrix artifacts, then document confirmed or likely defects without fixing them.

## Scope

In scope:
- Regex-driven Markdown parsing and artifact validation drift.
- Permissive impact schema/config validation and unknown-key handling.
- Test-only workspace registry failure toggles in production code paths.
- Repo-root detection, path containment, symlink/TOCTOU, destructive filesystem operations, and confirmation/report boundaries.
- Impact large-repo and large-diff scaling risks.

Out of scope:
- Broad refactors of monolithic MCP tool modules.
- Fixing parser, schema, filesystem, workspace, or impact behavior.
- Filing speculative security or performance notes as bugs without concrete evidence.

## Tasks

Create `.planning/phases/08-cross-cut-drift-and-regression-gaps/08-CONCERN-TRIAGE.md` with:

- A discovery-only boundary statement.

- Rows for `Regex-driven Markdown parsing`, `Deep-clone via JSON stringify/parse`, `Schema validation configured permissively`, `Test-only failure injection toggles`, `Repo-root detection is existence-based only`, `Path containment TOCTOU/symlink edges`, `Destructive filesystem operations`, `Network access in update checks`, `Prompt-boundary heuristics`, `Impact dependency scanning`, `Impact large diff processing`, and `Large artifact validation regex passes`.

#### Action

For each concern row, compare exact source behavior and tests:

- Markdown parsing: identify whether section/table regexes can silently parse the wrong artifact or mutate unintended content; cite tests that cover or miss the edge.
- Schema permissiveness: inspect impact config/report validation, unknown-key warnings, and schema failure handling; file only if unknown keys can create misleading output.
- Test-only toggles: inspect `BLUEPRINT_TEST_*` environment gates and decide whether production activation is plausible enough for a bug.
- Repo-root/path containment: inspect `ensureRepoRoot`, `ensurePathWithinRootSync`, and destructive `fs.rm` or `fs.rename` flows; file only with concrete destructive impact or contract violation.
- Impact scaling: inspect source traversal, diff parsing, lockfile handling, confidence/unknowns, and tests; file only if scaling limits can produce misleading confidence.
