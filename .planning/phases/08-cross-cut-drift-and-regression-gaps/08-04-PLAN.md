# Phase 8 Plan 04: Root-Cause Cluster And Related-Bug Linkage

## Goal

Link BPBUG-001 through BPBUG-004 and any new Phase 8 findings into practical root-cause clusters while preserving each defect's individual identity and repair path.

## Scope

In scope:
- `docs/bugs/INDEX.md` root-cause cluster section and Phase 8 bug inventory updates.
- `## Related Bugs` sections in BPBUG-001 through BPBUG-004 and any new Phase 8 bug reports.
- Duplicate classification review for any Phase 8 findings that overlap prior bugs.

Out of scope:
- Reclassifying severity/status as a repair-priority decision for Phase 9.
- Merging distinct defects that share only a broad root-cause family.
- Fixing the bugs or adding regression tests.

## Tasks

### Task 1: Derive cluster labels from bug and Phase 8 evidence

#### Read First

- `docs/bugs/INDEX.md`
- `docs/bugs/BPBUG-001-ship-undo-report-contracts-underconstrained.md`
- `docs/bugs/BPBUG-002-cleanup-lacks-behavioral-regression-coverage.md`
- `docs/bugs/BPBUG-003-mcp-tools-docs-stale-update-return-shapes.md`
- `docs/bugs/BPBUG-004-stale-built-bundle-omits-audit-fix-generated-assets.md`
- `.planning/phases/08-cross-cut-drift-and-regression-gaps/08-DRIFT-MATRIX.md`
- `.planning/phases/08-cross-cut-drift-and-regression-gaps/08-REGRESSION-GAPS.md`
- `.planning/phases/08-cross-cut-drift-and-regression-gaps/08-CONCERN-TRIAGE.md`

#### Action

Add or update a `## Root Cause Clusters` section in `docs/bugs/INDEX.md` with a table containing:

- `Cluster`
- `Bug IDs`
- `Shared Cause`
- `Preventive Repair Direction`

Seed the table with these cluster labels and exact starting assignments unless execution evidence justifies a different mapping:

- `under-specified contracts` -> BPBUG-001.
- `missing regression guards` -> BPBUG-002 and any Phase 8 high-risk test-gap bug.
- `docs/runtime synchronization` -> BPBUG-003 and any Phase 8 shared-doc drift bug.
- `generated-asset freshness` -> BPBUG-004 and any Phase 8 generated-asset recurrence bug.
- `cross-layer contract synchronization gaps` -> any Phase 8 drift bug that crosses docs, manifests, skills, source, tests, or `dist`.

#### Acceptance Criteria

- `rg -n "## Root Cause Clusters|under-specified contracts|missing regression guards|docs/runtime synchronization|generated-asset freshness|cross-layer contract synchronization gaps" docs/bugs/INDEX.md` finds the cluster section.
- `rg -n "BPBUG-001|BPBUG-002|BPBUG-003|BPBUG-004" docs/bugs/INDEX.md` finds the existing bugs inside bug rows or cluster assignments.
- The cluster table preserves the existing severity, confidence, surface, and status vocabularies.

### Task 2: Update related-bug sections without duplicating defects

#### Read First

- `docs/bugs/TEMPLATE.md`
- `docs/bugs/INDEX.md`
- `docs/bugs/BPBUG-*.md`

#### Action

For each BPBUG report affected by a root-cause cluster:

- Update `## Related Bugs` with a short bullet naming related BPBUG ids and the cluster relationship.
- Use `None known.` only when no related or clustered bug remains.
- Preserve each report's original expected behavior, actual behavior, evidence, verification steps, uncertainty, and no-fix sentence.
- Mark a report `duplicate` only if it shares the same defect and same repair path with a canonical bug; otherwise keep it separate and related.

#### Acceptance Criteria

- `rg -n "Related Bugs|root-cause|cluster|BPBUG-001|BPBUG-002|BPBUG-003|BPBUG-004" docs/bugs/BPBUG-*.md` finds related-bug context.
- `rg -n "status: duplicate" docs/bugs/BPBUG-*.md` returns no lines unless `docs/bugs/INDEX.md` also identifies the canonical duplicate target.
- `rg -n "No source, manifest, skill, test, generated asset, or runtime behavior fix was applied" docs/bugs/BPBUG-*.md` still finds the required no-fix sentence in every real bug report.
